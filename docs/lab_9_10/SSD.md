# System Specification Document (SSD)

## 1. Загальна інформація

- Система: `Petizoo`
- Тип: PHP MVC вебзастосунок для інтернет-магазину товарів для домашніх тварин
- Версія документа: `1.0`
- Дата: `2026-04-28`

## 2. Призначення системи

`Petizoo` забезпечує каталог товарів, керування кошиком, оформлення замовлень, профіль користувача та адміністративні функції для менеджменту товарів і замовлень.

## 3. Архітектурний стиль

Система реалізована за шаблоном `MVC`:

- `controllers/` містить контролери для бізнес-логіки та взаємодії з HTTP-запитом;
- `models/` містить моделі доступу до даних;
- `views/` містить шаблони інтерфейсу;
- `core/` містить інфраструктурний код: роутер, базовий контролер, роботу з БД, сесіями та шаблонами;
- `public/`, `css/`, `uploads/` містять статичні файли та завантажені зображення.

## 4. Склад системи

### 4.1 Ядро

- `core/Router.php` виконує розбір маршруту та виклик відповідного контролера/дії.
- `core/Core.php` надає доступ до глобальних компонентів.
- `core/DB.php` забезпечує роботу з базою даних через SQL-операції.
- `core/Session.php` керує сесією користувача.
- `core/Controller.php` є базовим класом для контролерів.

### 4.2 Основні модулі

- `Site`: головна сторінка та базові сторінки сайту.
- `Products`: каталог, фільтрація, перегляд товару, коментарі, обране.
- `Cart`: додавання, перегляд, збільшення, зменшення й видалення позицій кошика.
- `Checkout`: перевірка кошика, оформлення замовлення, сторінка успіху.
- `Users`: реєстрація, вхід, вихід.
- `Profile`: замовлення, обране, налаштування користувача.
- `Admin`: dashboard, товари, статуси замовлень, маркування популярних/акційних товарів.
- `Info`: статичні інформаційні сторінки.

## 5. Логіка маршрутизації

Маршрут формується як:

`/{controller}/{action}/{param1}/{param2}`

Правила:

- якщо маршрут порожній, використовується `site/index`;
- якщо action не задано, використовується `index`;
- router підтримує стандартні контролери та вкладені сценарії за іменуванням класів;
- параметри маршруту передаються в action як масив.

## 6. Функціональна специфікація

### 6.1 Каталог товарів

- відображення всіх товарів;
- фільтрація за призначенням для тварини (`cat`, `dog`, `both`);
- фільтрація за категорією та підкатегорією;
- пошук за назвою товару;
- перегляд сторінки окремого товару.

### 6.2 Обране

- доступне лише авторизованому користувачу;
- підтримує звичайний сценарій через redirect і AJAX-сценарій;
- якщо товар уже в обраному, повторний виклик видаляє його зі списку.

### 6.3 Кошик

- для авторизованого користувача зберігається в таблиці `cart_items`;
- для гостя зберігається в `$_SESSION['cart']`;
- підтримуються операції додавання, збільшення, зменшення й видалення товарів;
- окремо підтримується AJAX-додавання товару до кошика.

### 6.4 Оформлення замовлення

- checkout збирає товари з БД або сесії;
- перевіряє заповнення обов’язкових полів;
- валідує номер телефону;
- перевіряє достатність залишків;
- розраховує підсумкову вартість;
- створює замовлення й позиції замовлення;
- зменшує залишки товарів;
- очищає кошик після успішного оформлення.

### 6.5 Користувачі

- реєстрація за email і паролем;
- вхід із перевіркою пароля через `password_verify`;
- збереження авторизованого користувача в сесії;
- редагування окремих полів профілю.

### 6.6 Адміністрування

- доступ обмежується перевіркою логіна адміністратора;
- доступні перегляд статистики, товарів і замовлень;
- підтримується створення, редагування й видалення товарів;
- підтримується зміна статусів замовлень;
- підтримується встановлення ознак популярності та акційності.

## 7. Дані та сутності

На основі коду системи використовуються такі основні сутності:

- `users`: користувачі, облікові дані, профільні поля, ознака адміністратора;
- `products`: товари, опис, ціна, фото, залишок, підкатегорія, ознаки популярності/знижки;
- `categories`: категорії товарів;
- `subcategories`: підкатегорії товарів;
- `cart_items`: кошик авторизованого користувача;
- `favorites`: обрані товари користувача;
- `orders`: заголовок замовлення;
- `order_items`: позиції замовлення;
- `comments`: коментарі користувачів до товарів;
- `news`: новини.

## 8. Схема взаємодії компонентів

1. Користувач ініціює HTTP-запит.
2. `index.php` передає маршрут до `core/Router`.
3. Router знаходить контролер і action.
4. Контролер взаємодіє з моделями або напряму з `Core::get()->db`.
5. Результат передається у view.
6. View рендериться в HTML або action повертає JSON для AJAX-сценарію.

## 9. Специфікація інтерфейсів

### 9.1 Вебінтерфейс

- сторінки каталогу;
- сторінка товару;
- сторінки входу та реєстрації;
- кошик;
- checkout;
- профіль користувача;
- адміністративні сторінки;
- інформаційні сторінки.

### 9.2 HTTP/AJAX інтерфейс

У проєкті реалізовані ендпоінти, які повертають JSON і можуть бути описані в API-документації:

- `POST /cart/ajaxadd/{productId}`
- `POST /products/favoriteajax`

## 10. Правила валідації

- email при реєстрації повинен відповідати формату email;
- пароль проходить regex-перевірку на наявність літер, цифр і спецсимволів;
- підтвердження пароля повинно збігатися з основним паролем;
- ПІБ, телефон, місто та відділення/адреса обов’язкові для checkout;
- телефон повинен відповідати формату `+380XXXXXXXXX`;
- кількість товару в замовленні не може перевищувати залишок на складі;
- `product_id` для AJAX-операцій повинен бути коректним числом/ідентифікатором.

## 11. Вимоги до безпеки

- паролі зберігаються у хешованому вигляді;
- персональні дані користувача зберігаються на сервері;
- доступ до профілю та обраного вимагає авторизації;
- доступ до адміністративної панелі обмежений;
- сесійні дані використовуються для підтримки авторизації та кошика гостя.

## 12. Вимоги до продуктивності

- система повинна обслуговувати типові операції каталогу без помітної затримки для користувача;
- SQL-запити фільтрації товарів повинні виконуватися коректно для стандартного обсягу каталогу;
- AJAX-ендпоінти повинні повертати коротку JSON-відповідь для швидкого оновлення інтерфейсу.

## 13. Технічні залежності

- PHP-застосунок із власним MVC-ядром;
- MySQL база даних `petizoo`;
- веботочення `MAMP`;
- файлова система для зберігання зображень у `uploads/products/`.

## 14. Обмеження реалізації

- роль адміністратора визначається конкретним логіном, а не повноцінною RBAC-моделлю;
- частина логіки звертається напряму до `Core::get()->db`, без окремого сервісного шару;
- API не є повністю REST-орієнтованим, оскільки проєкт насамперед серверно-рендерений;
- дані кошика для гостя зберігаються в сесії, тому не синхронізуються між пристроями.

## 15. Критерії готовності

- документ описує фактичну архітектуру проєкту;
- перелічені основні модулі та бізнес-потоки;
- визначені сутності та правила валідації;
- зафіксовані HTTP-ендпоінти, які покриті Swagger/Postman документацією.
